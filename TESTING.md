# Тестирование и CI

Как фреймворк находит и запускает тесты. Рассчитано на большие монорепо с
множеством сервисов; для плоского репозитория работает так же без настройки.

## Модель в одном предложении

> Каждый каталог с `pyproject.toml`, содержащим `[project]`, — это **тестовое
> окружение**. CI ставит его зависимости через `uv` и запускает в нём `pytest`,
> который сам находит тесты внутри. Окружения тестируются параллельно и
> изолированно (каждое на своём раннере, в своём venv).

Что нужно знать разработчику — три пункта:
1. У сервиса есть `pyproject.toml` с `[project]`.
2. Тесты лежат где угодно **под** этим `pyproject.toml`.
3. Медленные/дорогие тесты помечены `@pytest.mark.heavy`.

Всё остальное — плумбинг в `.github/scripts/` и `.github/workflows/_tests.yml`,
читать его в обычной работе не нужно.

## Как обнаруживаются окружения

`.github/scripts/discover-envs.sh`:
- корневой `pyproject.toml` → окружение `.` (плоский проект / общие и e2e-тесты);
- по каждому `services/<имя>/` — **один** верхнеуровневый `pyproject.toml` с
  `[project]` (путь внутри сервиса не важен: `workdir/`, `backend/`, корень).

Имена сервисов нигде не зашиты. Сервис без python-манифеста (например JS-клиент)
в матрицу не попадает. Нет ни одного `[project]` — тесты просто скипаются (зелёно).

Если рядом с корневым окружением есть сервисные, корень запускается с
`--ignore=<каждый сервис>`, чтобы не собирать их тесты повторно. Если корневой
`pyproject.toml` единственный — он собирает тесты всего репозитория.

## feature vs PR

| | окружения | маркеры |
|---|---|---|
| push в `feature/**` | только **затронутые** изменениями ветки (affected) | `-m "not heavy"` |
| PR в `main` | **все** | все (включая `heavy`) |

Affected (`.github/scripts/affected-envs.sh`) сравнивает ветку с merge-base от
`main`: меняли файлы сервиса → тестируется этот сервис; меняли общий/корневой файл
→ перестраховка, гоняется всё; меняли только доки → ничего. Так `heavy`/e2e
по факту выполняются только на PR полным прогоном.

## Что должно быть в `pyproject.toml`

```toml
[project]
name = "auth-service"
version = "0.0.1"
requires-python = ">=3.11"
dependencies = ["fastapi", "sqlalchemy"]   # рантайм-зависимости сервиса

[build-system]                             # нужен, если тесты импортируют пакет
requires = ["hatchling"]                   # (src-layout). Иначе можно опустить и
build-backend = "hatchling.build"          # положиться на pythonpath/conftest.

[dependency-groups]
dev = ["pytest", "pytest-asyncio"]         # uv ставит dev-группу автоматически;
                                           # тест-зависимости держим здесь

[tool.pytest.ini_options]
testpaths = ["tests"]                      # pytest не уползёт за пределы окружения
markers = ["heavy: медленные/дорогие (e2e, load)"]
addopts = "--strict-markers"               # опечатка в маркере = ошибка, не молчание
# pythonpath = ["src"]                      # для src-layout без установки пакета
```

Обязателен только `[project]` (по нему идёт обнаружение). `testpaths` и
`--strict-markers` крайне желательны: они делают pytest источником правды о том,
что и где запускать, а не размазывают это по CI.

## Локальная проверка обнаружения

```bash
bash .github/scripts/discover-envs.sh        # какие окружения видит CI
```
