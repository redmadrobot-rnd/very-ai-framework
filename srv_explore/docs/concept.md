# srv-explore как сервис на сервере — окончательная концепция

Статус: реализовано и в PR #45, smoke-проверено на dev-хосте. srv-explore живёт как
изолированный сервис на сервере (remote MCP + readonly-агент за гардом), а из основного
проекта к нему подключаются по MCP с bearer-токеном — инженеру SSH к серверу не нужен.

## 1. Идея в одну строку

На сервере живёт настроенный нами **read-only агент за одним MCP-инструментом**
`srv_explore(task)`. Инженер из своего Claude Code шлёт задачу на естественном языке
— агент читает хост (файлы, логи, контейнеры, БД) **только на чтение**, крутит цикл
локально и возвращает findings-отчёт. Креды и enforcement живут на сервере, вне
машины инженера.

Почему это сильнее фазы 1: сейчас `guard.py` — хук на ноуте инженера, он его может
выключить (defense). На сервере гард и readonly-роль инженеру недоступны — они
становятся **настоящей границей**.

## 2. Ключевое решение: агент бежит НА ХОСТЕ, не в контейнере

Контейнер видит **своё** состояние: свой `df -h`, свой rootfs, свои версии пакетов/
драйверов, только свои процессы. Эксплореру нужно состояние **хоста** — иначе он
врёт. Плюс пути для чтения заранее неизвестны, предмонтировать `:ro` нечего — агенту
нужна свобода читать где угодно на хосте.

Вывод: сервис ставится на хост (systemd-юнит) под **непривилегированным read-only
OS-пользователем**. Это единственный компонент фреймворка, который намеренно живёт
**вне docker-пайплайна сервисов**. Цена — отдельная установка/апдейт; закрываем её
автоматизацией (setup-skill + шаг деплоя), а не ручными правками на сервере.

Отвергнутые варианты: контейнер+ro-mounts (не знаем путей, видит своё состояние);
контейнер с `--pid=host -v /:/host:ro` (видит хост, но пути через `/host`, трение с
journald/machine-id, слабее изоляция, docker.sock всё равно нужен — сложнее и менее
честно, чем просто хостовый процесс).

## 3. Компоненты политики read-only (всё в бандле srv_explore/)

- `guard.py` — PreToolUse-гард. На сервере он из «defense» становится границей.
- `profiles/*.json` — allowlist shell + SQL-профили. Сетевые команды (`curl`/`ssh`)
  выключены флагом `SRV_EXPLORE_NO_NETWORK` — на хосте с egress они = канал утечки при
  инъекции; локальное чтение (`df/ls/cat/journalctl/docker ps`) остаётся.
- `agent_prompt.md` — системный промпт агента (правила + формат отчёта).
- Рецепт readonly-роли БД (`profiles/postgres.json` → `readonly_recipe`).

Источник правды для гарда/профилей/промпта — один (каталог скилла); сервис их
переиспользует, не форкает.

## 4. Целевой сценарий использования

**Setup (один раз, на окружение).** Инженер запускает setup-skill фреймворка на
сервере. Скилл:
1. заводит **readonly-роль БД** по рецепту профиля, пароль кладёт в Environment Secret;
2. заводит **ro OS-пользователя** для сервиса;
3. раскладывает файлы сервиса (агент, гард, профили, MCP-обёртка) на хост;
4. ставит и стартует **systemd-юнит**;
5. настраивает **приватный доступ** к MCP-порту (не в открытый интернет — см. §6);
6. кладёт ключ API модели в Environment Secret.

**Запуск разведки (много раз).** Инженер в своём Claude Code:
1. вызывает MCP-инструмент `srv_explore(task, env)` с вопросом
   («почему сервис X отваливается по ночам?»);
2. серверный агент читает хост под readonly, гард режет всё не-read, аудит пишется
   на сервере;
3. агент возвращает findings-отчёт (вопрос → что проверил → находки → гипотеза →
   предложенное действие). Долгое копание — job-id + polling, не держим соединение.

**Если находка требует изменения.** Агент только предлагает. Выполняет — основной
агент сессии по одобрению человека, доступом инженера, живьём (модель из фазы 1 не
меняется). Серверный разведчик остаётся read-only.

## 5. Архитектура и компоненты

```
Машина инженера                      Сервер (dev/prod)
┌─────────────────┐                  ┌──────────────────────────────────────┐
│ Claude Code     │   MCP (authn,    │ systemd: srv-explore-mcp (ro OS-user) │
│  └─ вызывает    │──TLS, приват.───▶│  ├─ MCP-обёртка: tool srv_explore()   │
│     srv_explore │   сеть)          │  ├─ Claude Agent SDK (headless)       │
└─────────────────┘                  │  │   └─ субагент srv-explore + guard  │
                                     │  ├─ egress: только API модели         │
                                     │  ├─ аудит-лог (tamper-proof, на серв.) │
                                     │  └─ читает: FS/journald/docker.sock:ro,│
                                     │             БД через readonly-роль     │
                                     └──────────────────────────────────────┘
```

- **MCP-обёртка** — тонкий сервер, отдаёт `srv_explore(task)` + `srv_explore_status(job_id)`.
  Внутри дергает Claude Agent SDK headless (официально документированный паттерн
  «ephemeral»: задача на вход → результат; job-id+poll). Окружение (dev/prod) —
  идентичность инстанса, к ней привязан токен; клиент его не выбирает.
- **Agent SDK** — системный промпт из `agent_prompt.md`, `allowed_tools` +
  `permission_mode=dontAsk`, гард навешен PreToolUse-хуком (мост к `guard.py`).
  Рантайм — родной для стека, не самопал.
- **Идентичность** — ro OS-user + readonly-роль БД. Никаких write-грантов и superuser.
- **Авторизация к модели** — `CLAUDE_CODE_OAUTH_TOKEN` (агент = `claude` CLI внутри SDK).
  Деплой-шаг пишет его на сервер в `/etc/srv-explore/env`; юнит подхватывает через
  `EnvironmentFile=`. Ротация = смена Secret + передеплой. В репо/образ не попадает;
  инженер-пользователь его не видит (SSH у него нет); агенту сливать некуда (egress закрыт).
- **Аудит** — на сервере, инженер до него не дотягивается.

## 6. Безопасность (что реально держит)

- **Только чтение** — фундамент: readonly-роль БД (движок сам не даст write) + гард
  для shell + ro OS-user. Гард server-side = граница, не defense.
- **Ingress заперт** — открываем порт на сервере, за которым автономный агент.
  Authn по **статичному bearer-токену, который выдаёт и отзывает админ** (не джобы,
  не required-reviewer — универсально, работает и вне GitHub). Инженер держит только
  токен = «право спросить», SSH ему не нужен. Модель токена:
  - генерация: `python -c "import secrets; print('srvx_'+secrets.token_urlsafe(32))"`;
  - на сервере хранится **только `sha256`** токена в `/etc/srv-explore/tokens.json`
    (`{id, label, env, sha256, created}`) — утечка файла не раскрывает токены;
  - CLI сервиса: `srv-explore-token issue --label <кто> --env <dev|prod>` (печатает
    токен один раз, дописывает хэш) · `revoke <id>` · `list`; отзыв = убрать запись
    + reload юнита;
  - клиент: `claude mcp add --transport http srv-explore https://<host>/mcp
    --header "Authorization: Bearer <token>"`; сервер сверяет `sha256` на каждый
    запрос, нет совпадения/revoked → 401.
  - TLS — за reverse-proxy окружения; сервис слушает не в открытый интернет.
- **Egress заперт** — на хосте с интернетом инъекция в читаемых данных может увести
  их наружу. Поэтому: `curl`/сетевые команды **вон из профиля** этого сервиса; по
  желанию — egress-allowlist только на API модели. Тогда трифекта (private data +
  untrusted content + exfil) не сходится: write-пути нет, выход только в отчёт
  авторизованному инженеру, сеть закрыта.
- **Prompt-injection** — агент читает недоверенные данные (логи, строки БД). Ловим
  не парсером, а отсутствием путей наружу: no-write + no-egress + вывод только
  вызвавшему. Урок Supabase/Datadog: граница на роли СУБД, не на промпте/флаге.

## 7. Упаковка во фреймворк

Две изолированные части: **серверный бандл** и **коннектор**.

```
srv_explore/                         # ИЗОЛИРОВАННЫЙ БАНДЛ (ставится на хост отдельно)
  mcp_server.py                      #   remote MCP + агент
  guard.py  profiles/                #   политика read-only (единый источник правды)
  agent_prompt.md                    #   системный промпт агента
  token_store.py                     #   bearer-токены (CLI issue/revoke/list)
  install.sh  systemd/  requirements.txt
  README.md  docs/                   #   установка + концепция/ресёрч
.claude/skills/srv-explore/SKILL.md  # КОННЕКТОР: как подключиться из проекта
.claude/commands/srv-explore.md      #   /srv-explore <вопрос> → зовёт MCP-tool
tests/test_srv_explore_*.py          # guard / token / mcp (pytest)
```

- **Бандл самодостаточен:** `install.sh` копирует только `srv_explore/` (guard/profiles/
  prompt внутри) — ноль зависимостей от `.claude` основного проекта.
- **Деплой** — воркфлоу `deploy-srv-explore.yml` (`workflow_dispatch`, вне docker-
  пайплайна): scp `srv_explore/`, `install.sh`, инжект `CLAUDE_CODE_OAUTH_TOKEN` из
  Environment Secret, рестарт юнита. prod — под required reviewers Environment.
- **Коннектор** в `.claude/` только описывает подключение (`claude mcp add` + `/srv-explore`);
  ничего серверного/секретного не содержит.

## 8. Статус реализации

Сделано и в PR:
- **Серверный бандл** `srv_explore/`: remote MCP (`srv_explore`/`srv_explore_status`,
  job-id+poll) на Claude Agent SDK headless; каждая Bash-команда через `guard.py`
  PreToolUse-мостом; `agent_prompt.md` как системный промпт; `token_store` (bearer,
  `sha256`); `install.sh` + systemd; `curl`/`ssh` off (`SRV_EXPLORE_NO_NETWORK`).
- **Коннектор** `.claude/`: скилл (подключение) + `/srv-explore` (зовёт MCP-tool).
- **Деплой** `deploy-srv-explore.yml` (`workflow_dispatch`, вне docker-пайплайна).
- **Тесты** (pytest): guard, token, bearer-auth, мост к guard, `NO_NETWORK`.
- **Smoke на dev-хосте end-to-end:** сервис active, MCP `initialize` + bearer (401/200),
  реальный прогон агента (read → findings; write → отказ + предложение человеку).

Осознанно НЕ в этом PR (следующими): TLS в самом сервисе (пока за reverse-proxy);
автопровижининг readonly-роли БД (пока рецепт в профиле); типизированные tools вместо
части shell-поверхности.

## 9. Что взять как референс (не как зависимость)

- **Claude Agent SDK headless hosting** (code.claude.com/docs/en/agent-sdk/hosting +
  cookbook) — рантайм сервиса, родной для стека.
- **HolmesGPT** — образец дизайна read-only investigation-агента как сервиса
  (read-only by RBAC, задача→findings). Тяжёл и k8s-центричен — не тащим, смотрим.
- Прайор-арт и security-инциденты — `docs/research.md` (рядом).
